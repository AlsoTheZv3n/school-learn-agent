## E8 — Agent-Loop (LangGraph)

**Milestone:** M3 Learning Engine
**Quelldokument:** `docs/07-agent.md` (Detailspezifikation), übergeordnete Prinzipien in `docs/00-architecture.md`.

### Zielbild

Der pädagogische Loop wird als **explizite LangGraph-State-Machine** implementiert — der Agent als auditierbarer Zustandsgraph statt freilaufender LLM-Schleife. Ein Schüler-Turn fliesst deterministisch durch die Knoten:

```
        ┌─────────────── student answer / request ───────────────┐
        ▼                                                         │
   [route] ──► [retrieve] ──► (Frage?) ──► [assess] ──► [update_model] ──► END
        │                          │                                       
        └────────► (Erklärung/Hint?) ────────► [explain] (generativ, P2) ──► END
```

Am Ende dieses Epics läuft ein vollständiger `ANSWER`-Turn end-to-end gegen die Test-DB: Frage → Schülerantwort → kuratierte Bewertung → Mastery-Update im Learner-Modell. Der `explain`-Pfad liefert eine altersgerechte, anonymisierte Erklärung über den LLM-Client.

### Bezug zu den Kernprinzipien

- **P2 (kuratierte Bewertung):** `assess` bewertet ausschliesslich über den kuratierten Grader und Answer Key — niemals über LLM-Generierung. Generative Freiheit ist auf `explain` beschränkt.
- **P3 (Modell verbessert sich, nicht der Agent):** Das Verhalten des Agenten ist eine weitgehend deterministische Funktion des Learner-Modells; aktualisiert wird `learner_state` via `record_attempt`, nicht der Agent selbst.
- **P4 (keine PII extern):** Jeder externe LLM-Call läuft durch `scrub`; dem LLM werden ohnehin nur IDs/Skill-Keys übergeben.
- **P6 (Mensch im Loop):** Niedrige Grader-Konfidenz wird **nicht** automatisch in `learner_state` zementiert — der Pfad zum Lehrer-Review bleibt offen.
- **P7 (eine Plugin-Naht):** Der Agent-Loop bleibt **eine** Implementierung; einziger Adapterpunkt ist der über `get_grader(subject_key)` erreichte Grader.

### Tasks

- **AG-1 — LangGraph-State + Graph:** Definiert `TutorState` + `Intent` und verdrahtet den kompilierten Graphen (route → retrieve → conditional(assess→update_model | explain) → END).
- **AG-2 — Nodes:** Implementiert die fünf Knoten `route/retrieve/assess/update_model/explain`; `assess` kuratiert (P2), `update_model` schreibt via `record_attempt` und zementiert nur bei hoher Konfidenz (P3/P6), `explain` ist der generative Pfad (P4).
- **AG-3 — LLM-Client + Anonymisierung (`safety-critical`):** Implementiert `llm/anonymize.py` (scrub vor jedem externen Call), `llm/client.py` (Backend per `settings.llm_backend` umschaltbar) und `llm/prompts/` (z. B. `EXPLAIN_SYSTEM`).
